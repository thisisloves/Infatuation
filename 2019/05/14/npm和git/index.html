  • npm i module_name -S 等于 npm install module_name –save 等于 npm i -S module_name
  • 写入到 dependencies 对象
  • 用于开发环境，简单说就是开发时候会用到 生产环境也会用到

  • 会放到package-lock.json的dependencies中

  • npm i module_name -D 等于 npm install module_name –save-dev 等于 npm i -D module_name

  • devDependencies 对象
  • 用于开发环境，简单说就是开发时候会用到 生产环境用不到
  • 会放到package-lock.json的devDependencies中

  • 就是–save-dev 这样安装的包和版本号就保存在package.json的devDependencies里面

  • npm i module_name -g 全局安装

  • i 是install 的简写

git上传

  • 建立本地仓库
  • 待添加到本地仓库
  • 将项目的所有文件添加到仓库中
  • 输入本次的提交说明，提交暂存区中的更改的已跟踪文件，单引号内为说明内容
  • 关联远程仓库，添加后，远程库的名字就是 origin，这是 Git 默认的叫法，也可以改成别的，但是 origin 这个名字一看就知道是远程库。
  • 如果关联出现错误 fatal: remote origin already exists，则执行下列语句再进行关联
  • 代码合并
  • 把本地库的所有内容推送到远程库上
  • 如果在推送时出现错误 error:failed to push som refs to…….，则执行下列语句

git基本操作

  1. 查看配置信息 git config -l
  2. 初始化仓库(本地仓库) git init
  3. 克隆远程代码git clone url
  4. 拉取远程代码 git pull 相当于 git fetch 和git merge
  5. 从其他分支合并代码到当前分支 git merge branch-name
  6. 比较文件 git diff [filename]
  7. 添加文件 git add [.|filename]
  8. 提交文件 git commit -[a]m ‘备注信息’
  9. 查看仓库状态 git status
  10. 查看日志 git log
  11. 查看某个提交修改的内容 git show commitID
  12. 查看某个文件修改记录 git log -p filename

git分支管理

  • 查看分支
  1. git branch -all 全部分支
  2. git branch -remote 远程分支
  3. git branch -local 本地分支
  4. 创建分支 git branch branch_name
  5. 切换分支 git checkout branch_name
  6. 创建分支并切换到分支 git checkout -b new_branch exists_branch（默认为空，从master分支拉取代码）
  7. 删除分支，如果正在当前分支，则不能删除 git branch -d branch_name
  8. 删除远程分支 git push origin –delete branch_name
  9. 合并自分支代码，先切换到master分支 git checkout master 切换到主分支 git merge sub_branch 合并sub分支代码到主分支

  • 注意：如果在子分支修改了代码，未commit就切换到master，master也会显示文件被修改了

  • 推送分支代码到远程分支 git push origin master git push origin sub_branch 推送sub分支

  • 推送本地分支到远程，如果远程分支没有，则使用下面命令 git push –set-upstream origin invoice

代码回退

  1. 修改本地文件，还未add操作，注意：所有修改将丢失 git checkout – filename
  2. 添加新文件且执行了add，想返回未add状态，保留修改的把内容 git reset HEAD filename
  3. 如果文件被删除后，想要恢复源文件 git checkout HEAD – filename
  4. 文件执行了commit后，想回到上一个版本（log会被删除） git reset –soft commit_id
  5. 回退到指定版本，回到add后，commit前的代码 git reset commit_id
  6. 回退到指定版本修改后，回到修改后，add前的代码（默认–mixed，可省略） git reset –hard commit_id 回退到指定版本，回到修改前的代码
